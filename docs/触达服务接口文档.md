# 句子秒聘 · 触达服务接口文档

> 用于两个服务之间对齐边界与接口。定稿后表格服务据此实现「回填 / 转人工」接口，触达服务据此去表格化改造。

## 一、架构边界（两服务分工）

| 服务 | 职责 | 碰什么 |
|---|---|---|
| **表格管理服务** | 所有飞书表格操作：约面同步、简历识别、面评生成、字段校正联动 | 飞书多维表格 |
| **触达服务**（本服务） | 所有 MongoDB 操作 + 对外发消息：企微加好友、和候选人对话、约面沟通、日程创建 | MongoDB、秒回、大模型、飞书 IM/日历 |

- 两个服务**共用「句子秒聘」飞书应用**，部署在同一台公司服务器。
- 原则：**触达服务不再直接读写飞书表格**；一切「回填进度 / 标记转人工」都走表格服务提供的接口。

---

## 二、触达服务【对外提供】的接口（表格服务 → 触达服务）

### 2.1 POST /reach — 发起触达
表格服务在候选人约面信息就绪（进度表「一面面试官」+「一面时间」都填好）后调用，把候选人交给触达服务。

**请求体**
```json
{
  "dataId": "进度表记录的 record_id（回填/转人工时原样带回）",
  "phone": "18500000000",
  "name": "贾宏佳",
  "position": "运维工程师",
  "interviewer": "面试官姓名（与 HR 名录一致，用于建日程选人）",
  "interviewTime": "2026-07-21 14:00 或 毫秒时间戳"
}
```
| 字段 | 必填 | 说明 |
|---|---|---|
| dataId | 建议 | 进度表 record_id，回填/转人工回调时用它定位记录（缺则用 phone 兜底） |
| phone | 是 | 候选人手机号，触达用它加好友 |
| name / position | 否 | 用于话术占位符 {name} {position} |
| interviewer | 否 | 面试官姓名，建面试日程时按此在 HR 名录找邮箱 |
| interviewTime | 是 | 约面时间，欢迎语 {time} 与建日程用 |

**返回** `{ "ok": true, "taskId": "RT...", "status": "ADDING" }`

**触达服务后续自动流程**（无需表格服务干预）：
`建任务(Mongo) → 秒回加好友 → 好友通过 → 发带时间约面欢迎语 → 候选人回复经大模型识别意图 → 确认则建面试日程+回确认 → 需回填/转人工时回调表格服务`

## 三、触达服务【需要表格服务提供】的接口（触达服务 → 表格服务）

> 触达服务去表格化后，进度回填与转人工都调这两个接口。请表格服务实现，并给出 base URL（两服务同机，可 `http://127.0.0.1:<表格服务端口>`）。

### 3.1 POST /progress/backfill — 回填进度/备忘录
触达服务在关键节点调用，表格服务据此更新进度表（备忘录、状态、会议链接等）。

**请求体**
```json
{
  "dataId": "record_id（可空，空则用 phone 匹配）",
  "phone": "18500000000",
  "event": "ADD_SENT | CONFIRMED | WELCOMED | INTENT_ACCEPT | INTENT_RESCHEDULE | INTENT_REJECT | SCHEDULE_OK | ADD_FAILED | SEND_FAILED",
  "note": "追加到备忘录的中文进展，如：候选人确认面试时间【2026-07-21 14:00】",
  "status": "触达状态（可选，供表格服务写状态字段）",
  "interviewTime": "候选人最终确认/改约的时间（可选）",
  "meetingLink": "建日程后生成的飞书视频会议链接（可选）"
}
```
**返回** `{ "ok": true }`

**调用时机**：加好友结果、好友通过、候选人确认/改约/拒绝、约面日程创建成功、发送失败等。

### 3.2 POST /progress/handover — 转人工
触达服务判断需要真人介入时调用，表格服务：①把进度表「转人工」字段置为待处理 ②按需通知 HR。

**请求体**
```json
{
  "dataId": "record_id（可空，空则用 phone 匹配）",
  "phone": "18500000000",
  "reason": "REJECT | RESCHEDULE | KB_MISS | USER_REQUEST | SEND_FAILED",
  "reasonText": "转人工原因中文说明，如：候选人问到知识库未覆盖的问题",
  "candidateReply": "候选人原话（可选，供 HR 参考）"
}
```
**返回** `{ "ok": true }`

**触发场景（触达服务侧判定）**
| reason | 场景 |
|---|---|
| REJECT | 候选人明确拒绝面试 |
| RESCHEDULE | 候选人要改约，需 HR 协调新时间 |
| KB_MISS | 候选人问到知识库覆盖不了的问题（薪资细节、特殊诉求等） |
| USER_REQUEST | 候选人明确要求人工 / 找真人 |
| SEND_FAILED | 加好友或发消息失败，需人工兜底 |

## 四、数据表需新增字段（进度表）

| 字段名 | 类型 | 说明 |
|---|---|---|
| **转人工** | **是 / 否**（布尔/单选） | 触达服务 `/progress/handover` 触发后由表格服务置为「是」；HR 据此接手。只标记是否需人工，不分处理状态 |

（可另加「触达状态」「会议链接」字段，供 backfill 写入，非必需）

## 五、触达服务内部接口（秒回平台 → 触达服务，表格服务无需关心）

### POST /mh/callback 及 /mh/callback/*
秒回小组级回调统一入口。秒回把「好友通过 / 收到消息 / 发送结果」按 `base + 子路径` 推送，触达服务按 body 字段分发处理（好友通过存标识、消息交大模型对话、发送失败转人工）。回调地址在秒回小组控制台配置为 `http://ai-hr.juzibot.com/mh/callback`。

---

## 六、配置台（触达服务自带）
